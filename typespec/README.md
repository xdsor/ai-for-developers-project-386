# Meeting Booking Service

## Описание проекта

`Meeting Booking Service` - сервис бронирования встреч.

Встреча в домене называется бронированием (`Booking`). Чтобы бронирование появилось, гость должен открыть ссылку на событие (`Event`) и выбрать свободный временной слот (`TimeSlot`).

Событие (`Event`) описывает тематику встречи: название, описание и длительность в минутах. Пользователь (`User`) создает событие у себя в профиле, после чего сервис формирует публичную ссылку на это событие. Этой ссылкой можно делиться с любым количеством гостей.

Гость может попасть в сервис двумя способами:

- открыть ссылку на конкретное событие (`Event`) и сразу перейти к выбору свободного слота;
- открыть ссылку на пользователя (`User`) и увидеть все события, которые создал этот пользователь, а затем выбрать одно из них.

После выбора свободного слота гость создает бронирование (`Booking`). Отдельной сущности гостя в MVP нет: контактные данные гостя хранятся прямо в бронировании через `GuestContact`.

## MVP-ограничения

- В системе пока есть один пользователь.
- Сессии и авторизации пока нет.
- Доступные слоты формируются на 14 дней, начиная с текущей даты.
- В MVP пользователь считается доступным круглые сутки в рамках этого 14-дневного окна.
- Гость может создать бронирование только на свободный слот из этого окна.
- Нельзя создать два бронирования на пересекающиеся временные интервалы одного пользователя, даже если бронирования относятся к разным событиям.

## Доменные сущности

### `User`

Пользователь, который создает события и принимает бронирования.

В текущем MVP пользователь один, но сущность `User` остается в модели, потому что события и бронирования принадлежат владельцу.

Поля:

- `id: ResourceId` - внутренний идентификатор пользователя.
- `name: string` - имя пользователя.
- `slug: Slug` - публичный идентификатор пользователя для ссылок.
- `timeZone: IanaTimeZone` - часовой пояс пользователя для расчета текущей даты и 14-дневного окна.

### `Event`

Событие, то есть тематика встречи, которую можно забронировать.

Поля:

- `id: ResourceId` - внутренний идентификатор события.
- `ownerId: ResourceId` - идентификатор пользователя-владельца.
- `title: string` - название события.
- `description: string` - описание события.
- `durationMinutes: int32` - длительность встречи в минутах.
- `slug: Slug` - публичный идентификатор события для ссылок.

### `GuestContact`

Контактные данные гостя, который создает бронирование.

Отдельной сущности `Guest` в MVP нет.

Поля:

- `name: string` - имя гостя.
- `email: Email` - email гостя.

### `TimeSlot`

Свободный временной слот, вычисляемый для конкретного события.

`TimeSlot` не является основной хранимой сущностью домена. Это read model для ответа API: сервис строит слоты на основе длительности события, 14-дневного окна и уже существующих бронирований пользователя.

Поля:

- `startAt: utcDateTime` - начало слота.
- `endAt: utcDateTime` - конец слота.
- `available: true` - признак, что слот свободен.

### `Booking`

Бронирование, то есть встреча, созданная гостем.

Поля:

- `id: ResourceId` - внутренний идентификатор бронирования.
- `eventId: ResourceId` - идентификатор события.
- `ownerId: ResourceId` - идентификатор пользователя-владельца события.
- `guest: GuestContact` - контактные данные гостя.
- `startAt: utcDateTime` - начало бронирования.
- `endAt: utcDateTime` - конец бронирования.
- `createdAt: utcDateTime` - дата и время создания бронирования.

## Вспомогательные модели API

- `UserProfile` - публичный профиль пользователя вместе со списком его событий.
- `EventList` - список событий.
- `TimeSlotList` - список свободных слотов.
- `BookingList` - список бронирований.
- `CreateEventRequest` - запрос на создание события.
- `UpdateEventRequest` - запрос на обновление события.
- `CreateBookingRequest` - запрос на создание бронирования.
- `Error` - стандартная ошибка API.
- `ErrorCode` - код ошибки API.

## Минимальные ограничения моделей

В TypeSpec зафиксированы только простые контрактные ограничения, которые можно выразить через OpenAPI schema validation:

- `NonEmptyString` - строка с `minLength: 1`.
- `Slug` - непустая строка, без дополнительного regex-формата в MVP.
- `Email` - строка с `format: email`.
- `IanaTimeZone` - непустая строка, без перечисления всех IANA timezone в контракте.
- `PositiveDurationMinutes` - `int32` от `1` до `1440` включительно.

Эти ограничения используются в `User`, `Event`, `GuestContact`, `CreateEventRequest` и `UpdateEventRequest`.

Правила вроде `startAt < endAt`, `Booking.endAt = Booking.startAt + Event.durationMinutes`, проверки 14-дневного окна и отсутствия пересечений между бронированиями остаются бизнес-валидацией API, потому что зависят от текущей даты и состояния системы.

## Правила домена

- Публичная ссылка на пользователя строится через `User.slug`.
- Публичная ссылка на событие строится через `User.slug` и `Event.slug`.
- Слот для бронирования строится с учетом `Event.durationMinutes`.
- `Booking.endAt` вычисляется как `Booking.startAt + Event.durationMinutes`.
- Бронирование можно создать только в 14-дневном окне, начиная с текущей даты в `User.timeZone`.
- Новый `Booking` не должен пересекаться с уже существующими бронированиями того же `ownerId`.
- Проверка пересечений выполняется на уровне пользователя, а не события: пользователь не может быть на двух встречах одновременно.
- Если слот занят, API должно возвращать ошибку `SlotUnavailable`.
- Если слот вне 14-дневного окна, API должно возвращать ошибку `SlotOutsideBookingWindow`.
- Если интервал бронирования не соответствует длительности события, API должно возвращать ошибку `SlotDurationMismatch`.

## API-сценарии

### Публичные сценарии гостя

- `PublicUsers.readProfile` - получить публичный профиль пользователя и список его событий.
- `PublicUsers.listEvents` - получить список событий пользователя.
- `PublicUsers.readEvent` - получить конкретное событие по `userSlug` и `eventSlug`.
- `PublicBookings.listSlots` - получить свободные слоты события на ближайшие 14 дней.
- `PublicBookings.createBooking` - создать бронирование для свободного слота.

Публичные маршруты:

- `GET /users/{userSlug}`
- `GET /users/{userSlug}/events`
- `GET /users/{userSlug}/events/{eventSlug}`
- `GET /users/{userSlug}/events/{eventSlug}/slots`
- `POST /users/{userSlug}/events/{eventSlug}/bookings`

### Сценарии управления

В текущем MVP нет сессии и авторизации, но в TypeSpec контракте оставлены операции управления пользователем и его событиями.

- `AdminUsers.read` - получить пользователя по `userId`.
- `AdminEvents.list` - получить список событий пользователя.
- `AdminEvents.create` - создать событие пользователя.
- `AdminEvents.read` - получить событие пользователя.
- `AdminEvents.update` - обновить событие пользователя.
- `AdminEvents.delete` - удалить событие пользователя.
- `AdminBookings.list` - получить список бронирований пользователя по всем его событиям.

Админские маршруты:

- `GET /admin/users/{userId}`
- `GET /admin/users/{userId}/events`
- `POST /admin/users/{userId}/events`
- `GET /admin/users/{userId}/events/{eventId}`
- `PATCH /admin/users/{userId}/events/{eventId}`
- `DELETE /admin/users/{userId}/events/{eventId}`
- `GET /admin/users/{userId}/bookings`

## Структура TypeSpec

- `main.tsp` - входная точка сервиса, импорты и `@service`.
- `models.tsp` - доменные сущности, request/list модели и ошибки.
- `operations.tsp` - HTTP-интерфейсы и маршруты.

Сгенерированная OpenAPI-схема складывается в `tsp-output/schema/openapi.yaml`.
