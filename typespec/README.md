# Meeting Booking Service

## Описание проекта

`Meeting Booking Service` - сервис бронирования встреч.

Встреча в домене называется бронированием (`Booking`). Чтобы бронирование появилось, гость должен открыть ссылку на событие (`Event`) и выбрать свободный временной слот (`TimeSlot`).

Событие (`Event`) описывает тематику встречи: название, описание и длительность в минутах. Хост (`Host`) создает событие у себя в профиле, после чего сервис формирует публичную ссылку на это событие. Этой ссылкой можно делиться с любым количеством гостей.

Гость может попасть в сервис двумя способами:

- открыть ссылку на конкретное событие (`Event`) и сразу перейти к выбору свободного слота;
- открыть ссылку на хоста (`Host`) и увидеть все события, которые создал этот хост, а затем выбрать одно из них.

После выбора свободного слота гость создает бронирование (`Booking`). Отдельной сущности гостя в MVP нет: контактные данные гостя хранятся прямо в бронировании через `GuestContact`.

## MVP-ограничения

- В системе пока есть один хост.
- Сессии и авторизации пока нет.
- Доступные слоты формируются на 14 дней, начиная с текущей даты.
- В MVP хост считается доступным круглые сутки в рамках этого 14-дневного окна.
- Гость может создать бронирование только на свободный слот из этого окна.
- Нельзя создать два бронирования на пересекающиеся временные интервалы одного хоста, даже если бронирования относятся к разным событиям.

## Доменные сущности

### `Host`

Хост, который создает события и принимает бронирования.

В текущем MVP хост один, но сущность `Host` остается в модели, потому что события и бронирования принадлежат владельцу.

Поля:

- `id: ResourceId` - внутренний идентификатор хоста.
- `name: string` - имя хоста.
- `slug: Slug` - публичный идентификатор хоста для ссылок.
- `timeZone: IanaTimeZone` - часовой пояс хоста для расчета текущей даты и 14-дневного окна.

### `Event`

Событие, то есть тематика встречи, которую можно забронировать.

Поля:

- `id: ResourceId` - внутренний идентификатор события.
- `ownerId: ResourceId` - идентификатор хоста-владельца.
- `title: string` - название события.
- `description: string` - описание события.
- `durationMinutes: int32` - длительность встречи в минутах.
- `slug: Slug` - публичный идентификатор события для ссылок.

### `GuestContact`

Контактные данные гостя, который создает бронирование.

Отдельной сущности `Guest` в MVP нет.

Поля:

- `name: string` - имя гостя.
- `email: Email` - email гостя.

### `TimeSlot`

Свободный временной слот, вычисляемый для конкретного события.

`TimeSlot` не является основной хранимой сущностью домена. Это read model для ответа API: сервис строит слоты на основе длительности события, 14-дневного окна и уже существующих бронирований хоста.

Поля:

- `startAt: utcDateTime` - начало слота.
- `endAt: utcDateTime` - конец слота.

### `Booking`

Бронирование, то есть встреча, созданная гостем.

Поля:

- `id: ResourceId` - внутренний идентификатор бронирования.
- `eventId: ResourceId` - идентификатор события.
- `ownerId: ResourceId` - идентификатор хоста-владельца события.
- `guest: GuestContact` - контактные данные гостя.
- `startAt: utcDateTime` - начало бронирования.
- `endAt: utcDateTime` - конец бронирования.
- `createdAt: utcDateTime` - дата и время создания бронирования.

## Вспомогательные модели API

- `HostProfile` - публичный профиль хоста вместе со списком его событий.
- `EventBookingPage` - агрегированные данные для публичной страницы записи.
- `EventList` - список событий.
- `TimeSlotList` - список свободных слотов.
- `BookingList` - список бронирований.
- `CreateEventRequest` - запрос на создание события.
- `UpdateEventRequest` - запрос на обновление события.
- `CreateBookingRequest` - запрос на создание бронирования.
- `Error` - стандартная ошибка API.
- `ErrorCode` - код ошибки API.

## Минимальные ограничения моделей

В TypeSpec зафиксированы только простые контрактные ограничения, которые можно выразить через OpenAPI schema validation:

- `NonEmptyString` - строка с `minLength: 1`.
- `Slug` - непустая строка, без дополнительного regex-формата в MVP.
- `Email` - строка с `format: email`.
- `IanaTimeZone` - непустая строка, без перечисления всех IANA timezone в контракте.
- `PositiveDurationMinutes` - `int32` от `1` до `1440` включительно.

Эти ограничения используются в `Host`, `Event`, `GuestContact`, `CreateEventRequest` и `UpdateEventRequest`.

Правила вроде `startAt < endAt`, `Booking.endAt = Booking.startAt + Event.durationMinutes`, проверки 14-дневного окна и отсутствия пересечений между бронированиями остаются бизнес-валидацией API, потому что зависят от текущей даты и состояния системы.

## Правила домена

- Публичная ссылка на хоста строится через `Host.slug`.
- Публичная ссылка на событие строится через `Host.slug` и `Event.slug`.
- Слот для бронирования строится с учетом `Event.durationMinutes`.
- `Booking.endAt` вычисляется как `Booking.startAt + Event.durationMinutes`.
- Бронирование можно создать только в 14-дневном окне, начиная с текущей даты в `Host.timeZone`.
- Новый `Booking` не должен пересекаться с уже существующими бронированиями того же `ownerId`.
- Проверка пересечений выполняется на уровне хоста, а не события: хост не может быть на двух встречах одновременно.
- Если слот занят, API должно возвращать ошибку `SlotUnavailable`.
- Если слот вне 14-дневного окна, API должно возвращать ошибку `SlotOutsideBookingWindow`.
- Если интервал бронирования не соответствует длительности события, API должно возвращать ошибку `SlotDurationMismatch`.

## API-сценарии

### Публичные сценарии гостя

- `GuestHosts.readProfile` - получить публичный профиль хоста и список его событий.
- `GuestHosts.listEvents` - получить список событий хоста.
- `GuestHosts.readEvent` - получить конкретное событие по `hostSlug` и `eventSlug`.
- `GuestBookings.readBookingPage` - получить данные для публичной страницы записи на событие.
- `GuestBookings.listSlots` - получить свободные слоты события на ближайшие 14 дней.
- `GuestBookings.createBooking` - создать бронирование для свободного слота.

Публичные маршруты:

- `GET /hosts/{hostSlug}`
- `GET /hosts/{hostSlug}/events`
- `GET /hosts/{hostSlug}/events/{eventSlug}`
- `GET /hosts/{hostSlug}/events/{eventSlug}/booking-page`
- `GET /hosts/{hostSlug}/events/{eventSlug}/slots`
- `POST /hosts/{hostSlug}/events/{eventSlug}/bookings`

### Сценарии управления

В текущем MVP нет сессии и авторизации, но в TypeSpec контракте оставлены операции управления хостом и его событиями.

- `HostAccount.read` - получить хоста по `hostId`.
- `HostEvents.list` - получить список событий хоста.
- `HostEvents.create` - создать событие хоста.
- `HostEvents.read` - получить событие хоста.
- `HostEvents.update` - обновить событие хоста.
- `HostEvents.delete` - удалить событие хоста.
- `HostBookings.list` - получить список бронирований хоста по всем его событиям.

Маршруты управления:

- `GET /host/{hostId}`
- `GET /host/{hostId}/events`
- `POST /host/{hostId}/events`
- `GET /host/{hostId}/events/{eventId}`
- `PATCH /host/{hostId}/events/{eventId}`
- `DELETE /host/{hostId}/events/{eventId}`
- `GET /host/{hostId}/bookings`

## Структура TypeSpec

- `main.tsp` - входная точка сервиса, импорты и `@service`.
- `models.tsp` - доменные сущности, request/list модели и ошибки.
- `operations.tsp` - HTTP-интерфейсы и маршруты.

Сгенерированная OpenAPI-схема складывается в `tsp-output/schema/openapi.yaml`.
